# Migrations

Written by hand, not autogenerated — there's no live PostgreSQL
connection available in the environment these were built in for
`alembic revision --autogenerate` to inspect, so every migration here
was built by reading `app/models/*.py` directly instead. **None of this
has been run against a real database.** Test against a disposable/
staging Postgres instance before trusting it against anything with real
data — see the root README for the general "nothing here has been
proven correct by actually running it" caveat that applies to this
project throughout, and which matters more here than almost anywhere
else in the codebase.

## Which command to run depends on what you're starting from

**A genuinely fresh database** (nothing in it yet):
```
alembic upgrade head
```
Runs `0001_baseline` (creates every table), `0002_must_change_password`
(adds the one column baseline deliberately leaves out), then
`0003_permissions_and_roles` (adds `extra_permissions` and the four
newer role values). You end up with the full current schema either way.

**A database that already has every table and `must_change_password`**
— i.e. you deployed after that column was added but before
`extra_permissions`/the new roles existed:
```
alembic stamp 0002_must_change_password
alembic upgrade head
```

**A database from before either column existed** — the original
`create_all()`-only deployments:
```
alembic stamp 0001_baseline
alembic upgrade head
```

Both `stamp` commands tell Alembic "treat this revision as already
applied" *without* running its `upgrade()` (so it won't try to
`CREATE TABLE users` on a table that already exists and fail); the
`upgrade head` that follows only runs whatever comes after the stamped
point. If you're not sure which situation you're in: connect to the
database and check whether `users.must_change_password` already exists
as a column — if it does, use the second case; if `users` exists at all
but that column doesn't, use the third.

## Why `Base.metadata.create_all()` is still in `main.py`

It's harmless to leave running — it only creates tables that don't
exist yet, never alters ones that do, so it can't conflict with
Alembic or damage existing data. What it *can't* do, which is the
actual gap that led to this folder existing: add a new column to a
table that's already there. That's Alembic's job from here on. Treat
`create_all()` as fresh-install convenience only, not as "schema
changes work themselves out automatically" — they don't; a migration
needs to be written and actually run.

## Adding a new migration later

```
alembic revision -m "describe the change"
```
Then hand-write (or, with a real database connection available,
autogenerate via `alembic revision --autogenerate -m "..."` and review
the result — autogenerate is not perfect, especially around server
defaults and enum types) the `upgrade()`/`downgrade()` functions, set
`down_revision` to whatever the current latest revision id is, and test
it against a copy of a real database before running it anywhere that
matters.
